## User Story
Como motor matemático do Aura,
quero que o banco tenha os campos `percentual_contingencia`, `data_inicio_real` e `caminho_critico_baseline_dias` no TAP, e que os priors Bayesianos estejam na escala correta do novo denominador,
para que a Story 3.0-B possa implementar os denominadores corretos sem inconsistência de dados.

---

## Acceptance Criteria

### AC-1 — Campo `percentual_contingencia`
- [ ] Migration adiciona coluna `percentual_contingencia NUMERIC(5,2) DEFAULT NULL` em `projetos` ou `tap_dados`
- [ ] UI do TAP exibe slider 0–40% com label "Reserva de Contingência (%)"
- [ ] Valor default preenchido automaticamente pelo setor do projeto via `aura_setor_config`
- [ ] Se setor não mapeado: usa 10% com aviso amarelo: "Setor não mapeado — usando 10% como padrão. Ajuste se necessário."
- [ ] Campo persistido ao salvar TAP

### AC-2 — Campo `data_inicio_real`
- [ ] Migration adiciona coluna `data_inicio_real DATE DEFAULT NULL` em `projetos` ou `tap_dados`
- [ ] UI do TAP exibe campo de data "Data de Início Real" — nullable, não obrigatório
- [ ] Se nulo: sistema usa `prazo_inicio` do TAP com aviso amarelo: "Data de início real não informada — usando data planejada. Atualize quando o projeto iniciar."
- [ ] Campo persistido ao salvar TAP

### AC-3 — Campo `caminho_critico_baseline_dias`
- [ ] Migration adiciona coluna `caminho_critico_baseline_dias INTEGER DEFAULT NULL` em `projetos` ou `tap_dados`
- [ ] Campo preenchido automaticamente ao finalizar setup de tarefas (após CPM calculado pela primeira vez)
- [ ] Campo é **imutável após o primeiro preenchimento** — não sobrescrito em recálculos posteriores do CPM
- [ ] Se nulo: `recalcularTA()` usa `prazo_total` do TAP como fallback com warning no console

### AC-4 — Re-escalonamento dos Priors Bayesianos
- [ ] Migration aplica fator de ajuste nos `mated_medio` dos priors existentes:
  ```sql
  fator = 1.0 / (1.0 - percentual_contingencia_default / 100.0)
  -- construcao_civil: ×1.176 | tecnologia: ×1.111
  -- infraestrutura: ×1.176  | saude: ×1.111
  ```
- [ ] Apenas `tipo = 'prior'` é atualizado — empíricos não são tocados
- [ ] Migration inclui comentário com justificativa matemática da re-escalonamento
- [ ] `SELECT setor, COUNT(*), AVG(mated_medio) FROM aura_calibration_events WHERE tipo='prior' GROUP BY setor` retorna valores maiores que os anteriores (validação de sanidade)

### AC-5 — RLS e Segurança
- [ ] Novas colunas protegidas por RLS existente da tabela
- [ ] `caminho_critico_baseline_dias` somente escrito por `auth.uid()` do tenant do projeto

## Riscos
| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Migration altera `mated_medio` de forma irreversível — se fator errado, dados corrompidos | Alta | Alto | Fazer backup dos 12 priors antes de aplicar; incluir migration de rollback |
| `caminho_critico_baseline_dias` pode ser sobrescrito se lógica de imutabilidade falhar | Média | Alto | Usar `UPDATE ... WHERE caminho_critico_baseline_dias IS NULL` — nunca sobrescreve valor existente |
| Slider 0–40% pode confundir PM — diferença entre contingência e MR não é óbvia | Média | Médio | Tooltip explicativo: "Reserva de Contingência ≠ Management Reserve. MR não entra neste campo." |
