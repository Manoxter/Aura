## User Story
Como motor matemático do Aura,
quero que o Lado E seja calculado dinamicamente como `nTarefasAtual / nTarefasBaseline`,
para que o triângulo reflita scope creep real (adição de tarefas) ou redução de escopo durante a execução.

---

## Acceptance Criteria
- [x] AC-1: `gerarTrianguloCDT()` aceita `nTarefasAtual` e `nTarefasBaseline` como inputs
- [x] AC-2: `E = nTarefasAtual / nTarefasBaseline` quando ambos disponíveis e baseline > 0
- [x] AC-3: `E` clampado em 0.5 (proteção contra exclusão excessiva — Q2)
- [x] AC-4: Fallback `E = 1.0` quando baseline não registrado (primeiro uso)
- [x] AC-5: `escopo_ratio` exposto no CDTResult para display na UI (badge scope creep)
- [x] AC-6: CPM page registra `nTarefasBaseline` na primeira configuração do projeto
