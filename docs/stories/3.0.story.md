## Contexto

Debate produtivo do squad identificou dois erros fundamentais nos denominadores do Triângulo Atual:

1. **`lado_P`** usava `prazo_total` (caminho crítico + folgas) como denominador → mascara desvios reais
2. **`lado_O`** usava `orcamento_total` (incluindo reserva de contingência) como denominador → gordura artificial desde o início

Além da correção dos denominadores, o squad definiu um **Sistema de Zonas** completo para o CEt e para todos os dashboards do produto, com linguagem visual unificada e mecanismo de governança para casos de inviabilidade financeira.

## Decisões Canônicas do Squad (D1–D12)

### Grupo A — Denominadores CDT

| ID | Decisão |
|----|---------|
| D1 | `lado_P = duracao_acumulada / duracao_caminho_critico_baseline` — onde `duracao_acumulada` = dias corridos desde `data_inicio_real` do projeto (calendário) |
| D2 | `lado_O = custo_acumulado / orcamento_operacional` |
| D3 | TAP recebe campo `percentual_contingencia` (default por setor, range 0–40%) |
| D4 | Management Reserve NÃO entra no modelo CDT |
| D5 | `orcamento_operacional = orcamento_total × (1 − percentual_contingencia / 100)` |
| D6 | `duracao_caminho_critico` calculado automaticamente via CPM (tarefas com `critica: true`) |
| D7 | Default paramétrico por setor quando contingência não declarada: Construção=15%, TI=10%, Infraestrutura=15%, Saúde=10%. "Serviços" removido — não há benchmark confiável. Fallback global 10% com aviso explícito na UI do TAP |
| D7a | `duracao_caminho_critico_baseline_dias` registrado como snapshot imutável no TAP no momento da criação do projeto — denominador de P nunca muda retroativamente |
| D7b | `data_inicio_real` campo nullable no TAP — PM preenche quando projeto efetivamente inicia. Fallback: `prazo_inicio` com aviso na UI |
| D7c | TA usa posição (`dias_corridos / caminho_critico`) — `execution.ts`. TM usa velocidade (coef. burndown) — `math.ts`. São triângulos distintos com fórmulas de P distintas. TODO incorreto em `execution.ts:110` removido em 3.0-B |

### Grupo B — Sistema de Zonas (CEt + Dashboards)

| ID | Zona | Prazo | Custo | Ação do sistema |
|----|------|-------|-------|-----------------|
| D8 | **Verde** | ≤ caminho crítico | ≤ orçamento operacional | Decisão normal |

| D12 | **Nula** | — | além do orçamento total | Impossível — bloqueado por padrão |

## Acceptance Criteria

### AC1 — TAP: campo percentual_contingencia
- [ ] Campo `percentual_contingencia` adicionado ao schema (migration)
- [ ] UI do TAP com slider 0–40%, default por setor (Construção=15%, TI=10%, Serviços=12%)
- [ ] Campo obrigatório antes de liberar o motor matemático

### AC2 — Denominadores corretos em recalcularTA()
- [ ] `lado_P` usa `dias_corridos / duracao_caminho_critico_baseline_dias`
  - `dias_corridos = data_atual − data_inicio_real` (calendário)
  - Fallback: `data_inicio_real` nulo → usa `prazo_inicio` do TAP com warning
- [ ] `lado_O` usa `orcamento_operacional = orcamento_total × (1 − pct/100)`
- [ ] TODO incorreto em `execution.ts:110` removido (P não usa burndown ES/EF — esse é o TM)
- [ ] Fallback: se `duracao_caminho_critico_baseline_dias` nulo, usa `prazo_total` com warning
- [ ] Fallback: se `percentual_contingencia` ausente, usa default setorial de `aura_setor_config`
- [ ] `math.ts:556` fallback `contingenciaPct || 10` substituído por lookup setorial

### AC3 — Sistema de Zonas no CEt
- [ ] Função `classificarCandidatoCEt(candidato, limites)` em **`zones.ts`** (novo arquivo) retorna `{zona, prazo_real, custo_real, consequencias}`
  - Não em `cet.ts` nem em `cet-dimension.ts` — arquivo dedicado para zonas operacionais
- [ ] `cet-dimension.ts` mantido para fitting geométrico com `faixa_contingencia` renomeada para `faixa_ajuste` (elimina ambiguidade com contingência financeira)
- [ ] Zona Nula bloqueia exibição por padrão
- [ ] Unlock de Zona Nula requer `declaracaoAporteExterno: {texto, autor, timestamp}`
- [ ] Tabela de candidatos exibe: `+X dias`, `+R$ Y`, zona colorida

### AC4 — Visualização: Triângulo com Zonas
- [ ] TrianglePlotter renderiza faixas amarela (20% opacidade) e vermelha (15%) nos eixos P e O
- [ ] Triângulo atual pulsa suavemente quando em Zona Vermelha (CSS animation)
- [ ] Zona Nula não renderiza — canvas termina no teto absoluto

### AC5 — Visualização: Gráficos de Função
- [ ] Função Prazo (burndown) exibe banda de contingência de prazo em amarelo
- [ ] Função Custo (Curva S) exibe banda de contingência de custo em amarelo
- [ ] Linha horizontal no teto absoluto (vermelha, tracejada)

### AC6 — Semáforo no Header
- [ ] Componente `ZonaSemaforo` no header do motor: 🟢🟡🔴⚫
- [ ] Tooltip ao hover explica a zona e o que significa

### AC7 — Velocidade de Degradação
- [ ] Engine calcula `delta_distancia_NVO` entre último e penúltimo cálculo
- [ ] Se tendência de piora, exibe alerta: "Ritmo atual → Zona Amarela em ~X dias"
- [ ] Cálculo baseado em regressão linear dos últimos 5 pontos de distância_NVO

### AC8 — Histórico de Zonas
- [ ] Coluna `zona_resultado` em `progresso_tarefas` ou tabela auxiliar `historico_zonas`
- [ ] Componente `ZonaTimeline` exibe linha do tempo semana a semana
- [ ] Cores correspondentes às zonas no timeline

## Impacto no Pipeline de Processamento v2.0

O Passo 3 (Eixo de Dimensão — Zonamento CEt) já existe no pipeline. Esta story **substitui e expande** aquele passo:

```
PASSO 3 (ATUALIZADO) — ZONAMENTO CDT
  • Calcular orcamento_operacional = orcamento_total × (1 − pct_contingencia)
  • Calcular duracao_caminho_critico via CPM
  • Para cada candidato CEt:
      → classificar em Verde/Amarela/Vermelha/Cinza/Nula
      → desnormalizar: converter lados em dias e R$
      → calcular consequências
  • Calcular velocidade de degradação (derivada NVO)
  • Emitir alerta preditivo se tendência de piora
```

## Dev Notes

- `duracao_caminho_critico_baseline_dias` vem de `calculateCPMLocal()` em `cpm.ts` — snapshot imutável no TAP
- `duracao_acumulada` = `data_atual − data_inicio_real` em dias calendário (não dias úteis, não CPM EF)
- `data_inicio_real` nullable → fallback `prazo_inicio` do TAP + warning na UI
- `percentual_contingencia` nullable → fallback lookup em `aura_setor_config` por setor do projeto
- Setor "Serviços" não existe no banco — fallback global 10% com aviso explícito na UI
- `execution.ts` lado P: remover TODO "Story 1.3" — TA usa posição, TM usa velocidade (arquivos diferentes)
- `cet-dimension.ts` é para fitting geométrico (limites fixos 1.10/1.25) — não alterar os limites
- `zones.ts` (novo) é para zonas operacionais (limites dinâmicos por `percentual_contingencia`)
- O pulso CSS deve ser `animation: pulse 2s ease-in-out infinite` — não agressivo
- Zona Nula: nunca exibir como opção, só exibir mensagem de bloqueio
- Velocidade de degradação: janela mínima de 3 pontos, máximo 10 pontos históricos
- Re-escalonamento priors: migration aplica fator `1/(1−pct)` por setor antes de 3.0-B alterar o engine
