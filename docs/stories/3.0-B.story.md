## User Story
Como motor matemático do Aura,
quero que `recalcularTA()` use `dias_corridos / caminho_critico_baseline` para o Lado P e `custo_acumulado / orcamento_operacional` para o Lado O,
para que o Triângulo Atual reflita a realidade do projeto sem distorção de denominadores inflados.

---

## Acceptance Criteria

### AC-1 — Lado P corrigido
- [ ] `lado_P = dias_corridos / caminho_critico_baseline_dias`
  - `dias_corridos = DATEDIFF(data_atual, data_inicio_real)` em dias calendário
  - Fallback: `data_inicio_real` nulo → usa `prazo_inicio` do TAP + `console.warn()`
  - Fallback: `caminho_critico_baseline_dias` nulo → usa `prazo_total` do TAP + `console.warn()`
- [ ] TODO incorreto em `execution.ts:110` removido — comentário substituído por explicação da fórmula real
- [ ] Lado P pode ser > 1.0 (projeto atrasado além do caminho crítico) — não clampar em 1.0

### AC-2 — Lado O corrigido
- [ ] `orcamento_operacional = orcamento_total × (1 − percentual_contingencia / 100)`
- [ ] `lado_O = custo_acumulado / orcamento_operacional`
- [ ] Fallback: `percentual_contingencia` nulo → lookup em `aura_setor_config` pelo setor do projeto
- [ ] Fallback final: 10% global se setor não mapeado + `console.warn()`
- [ ] `math.ts:556` — fallback `contingenciaPct || 10` substituído por `getDefaultContingencia(setor)` de `aura_setor_config`

### AC-3 — Testes reescritos
- [ ] `execution.test.ts` reescrito: testes existentes que testavam `orcamento_base` atualizados para `orcamento_operacional`
- [ ] Novos testes adicionados:
  - Projeto com `percentual_contingencia=15%`: `orcamento_operacional = total × 0.85`
  - Projeto com `data_inicio_real` nulo: usa `prazo_inicio` + warning
  - Projeto com `caminho_critico_baseline_dias` nulo: usa `prazo_total` + warning
  - Lado P > 1.0 permitido (projeto atrasado)
- [ ] Todos os testes passam (mínimo 190/190)

### AC-4 — Sem regressão
- [ ] Lado E inalterado (`media percentual_avanco`)
- [ ] CEt e rollback inalterados
- [ ] TypeCheck: 0 erros novos
