## User Story
Como banco de dados do Aura,
quero as 4 novas tabelas criadas com RLS e índices,
para que o motor de execução, calibração e histórico funcionem corretamente.

## Acceptance Criteria
- [x] AC-1: Tabela `triangulo_matriz_versoes` criada com campos: id, projeto_id, versao, area_baseline, lados JSONB, motivo, criado_em
- [x] AC-2: Tabela `progresso_tarefas` criada: id, tarefa_id, percentual_avanco, registrado_em, registrado_por
- [x] AC-3: Tabela `decisoes_mated` criada: id, projeto_id, descricao, parametros_numericos JSONB, distancia_nvo, zona_resultado, impacto_area_percent, criado_em
- [x] AC-4: Tabela `aura_calibration_events` criada: id, projeto_id, setor, area_final, mated_medio, sdo_score, arquivado_em, modo (normal|invertido)
- [x] AC-5: RLS ativo em todas as 4 tabelas (isolamento por tenant)
- [x] AC-6: Índices: projeto_id em todas, tarefa_id em progresso_tarefas
- [x] AC-7: Migration com rollback testado localmente
- [x] AC-8: `supabase migration list` mostra migration aplicada

## Escopo
**IN:** DDL das 4 tabelas (`triangulo_matriz_versoes`, `progresso_tarefas`, `decisoes_mated`, `aura_calibration_events`), RLS por tenant em todas, índices de performance (projeto_id, tarefa_id), rollback testado localmente.
**OUT:** Não inclui RLS avançado com isolamento granular por operação (SELECT/INSERT/UPDATE/DELETE separados — isso é Story 8.11), não cobre seed de dados iniciais, não implementa triggers (`updated_at` — isso é Story 8.3).

## Estimativa

## Riscos
| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Migration aplicada em branch de desenvolvimento pode divergir do schema de produção se houver migrations pendentes não aplicadas — `supabase db push` pode falhar por conflito | Média | Alto | Executar `supabase migration list` antes de criar a migration; garantir branch dev está sincronizado com produção |
| RLS básico na Story 8.1 pode usar `auth.uid()` direto (ineficiente) — Story 8.5 corrige depois, mas cria debt temporário de performance | Alta | Baixo | Implementar já com `(SELECT auth.uid())` desde o início, seguindo o padrão da Story 8.5 |
