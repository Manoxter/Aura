## User Story
Como sistema multi-tenant do Aura,
quero que as 4 novas tabelas tenham políticas RLS completas (SELECT/INSERT/UPDATE/DELETE),
para que o isolamento por tenant seja total e a Story 8.1 esteja completa.

## Acceptance Criteria
- [x] AC-1: `triangulo_matriz_versoes`: SELECT/INSERT/UPDATE/DELETE isolados por `projeto_id → projetos.tenant_id`
- [x] AC-2: `progresso_tarefas`: SELECT/INSERT/UPDATE/DELETE isolados por `tarefa_id → tarefas.projeto_id → projetos.tenant_id`
- [x] AC-3: `decisoes_mated`: SELECT/INSERT/UPDATE/DELETE isolados por `projeto_id → projetos.tenant_id`
- [x] AC-4: `aura_calibration_events`: SELECT/INSERT/UPDATE/DELETE isolados por `projeto_id → projetos.tenant_id`
- [x] AC-5: RLS habilitado (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY`) em todas as 4
- [x] AC-6: Todas as policies usam `(SELECT auth.uid())` (padrão otimizado da Story 8.5)
- [x] AC-7: Teste: user_A não vê registros de calibração do user_B

## Tasks
- [ ] 1. Criar arquivo `supabase/migrations/YYYYMMDD_rls_novas_tabelas.sql`
- [ ] 2. Escrever helper function ou policy para lookup de tenant via JOIN em projetos
- [ ] 3. Para cada tabela: ENABLE RLS + 4 policies (SELECT, INSERT, UPDATE, DELETE)
- [ ] 4. Usar padrão: `EXISTS (SELECT 1 FROM projetos p WHERE p.id = {tabela}.projeto_id AND p.tenant_id = (SELECT auth.uid()))`
- [ ] 5. Para `progresso_tarefas`: join duplo via tarefas → projetos
- [ ] 6. Testar cross-tenant isolation com usuários distintos
- [ ] 7. Aplicar via `supabase db push`

## Escopo
**IN:** ENABLE ROW LEVEL SECURITY nas 4 tabelas novas, 16 policies granulares (SELECT/INSERT/UPDATE/DELETE × 4 tabelas) com `(SELECT auth.uid())`, lookup de tenant via JOIN em projetos, join duplo em `progresso_tarefas` via tarefas→projetos, teste cross-tenant.
**OUT:** Não inclui policies para service_role (acesso admin irrestrito é comportamento padrão Supabase), não cobre tabelas anteriores (escopo apenas das 4 novas — tabelas antigas são Story 8.5), não implementa policies de leitura compartilhada (ex: viewer com read-only).

## Riscos
| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Policy INSERT em `progresso_tarefas` requer join tarefas→projetos→tenant_id — query complexa pode ter performance ruim se não houver índice em `tarefas.projeto_id` | Alta | Médio | Verificar existência do índice em `tarefas.projeto_id` (criado na Story 8.1); criar se ausente antes de ativar RLS |
| `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` sem policies causa lock-out total — se migration for parcial (ENABLE sem políticas), nenhum usuário acessa a tabela | Alta | Alto | Garantir que ENABLE RLS e criação de todas as políticas estejam na mesma transação atômica |
