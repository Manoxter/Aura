## Respostas ao Architect

**1. Estrategia de ID:** UUID puro em todas as tabelas. O `id_string` (T01, T02) deve ser campo separado de display, nao PK. A TAP page e a EAP page devem ambas gerar UUIDs via `crypto.randomUUID()`.

**2. RLS Policies:** Por `tenant_id` com lookup via `auth.uid()`:
```sql
CREATE POLICY tenant_rls ON projetos
  FOR ALL USING (
    tenant_id IN (
      SELECT id FROM tenants WHERE owner_id = auth.uid()
    )
  );
```

**3. Cascading:** Sim, recomendo:
- `projetos` ON DELETE CASCADE → `eap_nodes`, `tarefas`, `orcamentos`, `marcos`, `funcoes_compressao`, `historico_projeto`
- `eap_nodes` **sem** cascade para `tarefas` (leaf node deletion deve ser explicita)

## Recomendacoes — Ordem de Resolucao

1. **Imediato:** DB1 (RLS) + DB5 (indices) — seguranca e performance baseline
2. **Sprint 1:** DB2 (padronizar IDs) + P5 (campo canonico duracao_estimada)
3. **Sprint 2:** DB9 + DB11 + DB12 — integridade referencial
4. **Backlog:** DB13 — triggers automaticos
